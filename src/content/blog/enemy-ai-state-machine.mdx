When the enemy's Chase→Patrol transition started misfiring because two branches shared a variable, I knew the update loop had to go.

## The Problem

The first version of the Hide & Seek enemy — the pattern I later carried into Relic Rush — was a single `Update()` method with a chain of `if/else` blocks. Patrol logic, detection checks, chase behaviour, and navigation calls were all in one place. Adding a search phase or alert scan meant the method grew — and changing any branch required reading the entire function to understand what else it might affect.

When I added the Search state, the Chase→Patrol transition started firing incorrectly because both branches read from the same `_lastKnownPosition` variable. Fixing one broke the other. That was the moment I stopped adding to the function and started replacing it.

The problem isn't just readability. A monolithic update loop has hidden coupling: the `chase` branch reads variables set by the `patrol` branch, transition conditions are scattered across the function, and there's no clear moment where one behaviour ends and another begins. Any new state has to understand all the other states to slot in correctly.

## The IState Interface

The core problem with the monolithic loop was shared mutable state — two branches reading and writing the same variables with no clear ownership. Extracting each behaviour into its own class removes that: each state owns its variables, and no other state can touch them.

The solution was to extract each behaviour into its own class implementing a three-method interface. Each state owns its entry logic, its per-frame update, and its exit cleanup — nothing more.

```cs
// Infrastructure/Interfaces/IState.cs
public interface IState
{
    void Enter();
    void Tick();
    void Exit();
}
```

`Enter()` runs once when a state becomes active — a good place to set NavMesh destinations, start animations, or reset local timers. `Tick()` runs every frame from the owner's `Update()`. `Exit()` cleans up before the next state takes over.

`BaseState` is an abstract class implementing `IState` with empty default methods — concrete states only override what they need.

A concrete state looks like this:

```cs
// AI/Enemy/States/EnemyPatrolState.cs
public class EnemyPatrolState : BaseState
{
    readonly EnemyController _enemy;
    int _waypointIndex;

    public EnemyPatrolState(EnemyController enemy)
    {
        _enemy = enemy;
    }

    public override void Enter()
    {
        _enemy.Navigation.SetSpeed(_enemy.Data.patrolSpeed);
        MoveToNextWaypoint();
    }

    public override void Tick()
    {
        if (_enemy.Navigation.IsAtDestination)
            MoveToNextWaypoint();

        if (_enemy.SuspicionMeter.State == SeekState.Alert)
            _enemy.ChangeState(new EnemyAlertState(_enemy));
    }

    void MoveToNextWaypoint()
    {
        var wp = _enemy.Waypoints[_waypointIndex % _enemy.Waypoints.Length];
        _enemy.Navigation.SetDestination(wp.position);
        _waypointIndex++;
    }
}
```

`SuspicionMeter` tracks how aware the enemy is of the player — `SeekState.Alert` means it has confirmed the player's position and is ready to chase. Waypoints are serialized `Transform` references on `EnemyController`, set in the Unity Inspector per-enemy.

Adding a new state means writing one new class. Nothing else changes.

## State Transitions

`EnemyController` is a passive hub: it holds references to subsystems and passes itself to each state — states decide what to do with those references. `StateMachine` manages the active state and drives transitions:

```cs
// Infrastructure/StateMachine/StateMachine.cs
public class StateMachine
{
    IState _currentState;

    public void ChangeState(IState newState)
    {
        _currentState?.Exit();
        _currentState = newState;
        _currentState?.Enter();
    }

    public void Tick() => _currentState?.Tick();
}
```

`EnemyController` owns one `StateMachine`, calls `Tick()` from `Update()`, and exposes a `ChangeState(IState)` wrapper so states can trigger transitions. The machine doesn't know what the states are — it only knows the `IState` interface.

## What I Learned

The biggest practical benefit wasn't cleanliness — it was **testability**. Each state is a plain C# class with no MonoBehaviour dependency. I can instantiate `EnemyPatrolState` in a unit test, feed it a mock `EnemyController`, and verify that it transitions when suspicion crosses the threshold. With a monolithic method, there's no clean seam to inject a mock — the logic and the dependencies are fused.

The thing I'd do differently: **pre-allocate states at `Awake()`** rather than calling `new EnemyChaseState(this)` on every transition. Each call allocates a new heap object. With six states and frequent transitions, this generates GC pressure. Creating all six states once at startup and passing them by reference eliminates that entirely. I tackle a related allocation problem differently in the [object pooling post](/blog/object-pooling-endless-runner).

{/* IMAGE: Unity Profiler memory view comparing GC allocations before and after pre-allocating enemy AI states
    Alt: "Unity Profiler memory view comparing GC allocations before and after pre-allocating enemy AI states" */}

**State as the unit of change.** When requirements changed — adding a `Search` state between `Chase` and `Patrol` — the existing states were untouched. The cost of change stayed local to the new file. That's the property worth naming: not cleanliness, but isolation.

**`EnemyController` as a contract, not a god object.** Passing the full controller into each state felt wrong at first. What stopped it from becoming a god object is that states only *read* from it — they don't *write* to shared fields. That discipline has to be maintained manually; the compiler won't enforce it.

The pattern scales — every new behaviour is a new file, not a new risk to existing code.
